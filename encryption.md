
Этот документ подробно описывает, как в проекте устроены:

- подключение клиента к серверу;
- обмен публичными ключами;
- шифрование и отправка сообщений;
- расшифрование на стороне получателя.


## 1) Роли и доверенная модель

В системе есть 2 типа участников:

- **Клиент** — генерирует ключи, шифрует и расшифровывает сообщения.
- **Сервер** — маршрутизирует пакеты между клиентами.

Сервер:

- знает никнеймы пользователей;
- хранит их публичные ключи;
- **не хранит** приватные ключи;
- **не расшифровывает** сообщения.

Важно: в текущем MVP сервер не проверяет подлинность публичных ключей через PKI/сертификаты, поэтому защита от подмены ключа не полная (см. раздел «Ограничения»).

## 2) Криптопримитивы

Используется библиотека `cryptography`.

### 2.1 Асимметричная криптография (RSA)

- Алгоритм: **RSA 2048**
- Назначение:
  - у каждого клиента есть постоянная (на сессию) пара ключей:
    - приватный ключ (только у клиента),
    - публичный ключ (можно передавать через сервер).
  - RSA используется для шифрования **сеансового симметричного ключа**.

В коде:

- генерация пары: `generate_rsa_keypair()`
- упаковка ключей в PEM (текстовый формат)

### 2.2 Симметричная криптография (AES-GCM)

- Алгоритм: **AES-256-GCM**
- Назначение:
  - шифрует сам текст сообщения.
- Параметры:
  - одноразовый `nonce` длиной 12 байт;
  - случайный AES-ключ 256 бит на **каждое сообщение**.

Почему гибридная схема:

- RSA неэффективен для больших данных;
- AES-GCM быстрый и даёт аутентифицированное шифрование.

Итого:

1. генерируется случайный AES-ключ;
2. сообщение шифруется через AES-GCM;
3. AES-ключ шифруется RSA-публичным ключом получателя;
4. отправляется связка `{encrypted_key, nonce, ciphertext}`.

## 3) Формат транспорта и пакетов

Сетевой транспорт — TCP.

На уровне протокола используется **framing**:

1. 4 байта (`!I`, big-endian) — длина JSON-пакета;
2. затем JSON-пакет в UTF-8.

Это защищает от «склейки»/«дробления» сообщений в TCP-потоке.

Каждый пакет содержит:

- `version` (сейчас `1`)
- `type` (тип сообщения)
- поля payload в зависимости от типа.

Пакеты валидируются функцией `validate_packet()`.

## 4) Подключение и регистрация клиента

После TCP-подключения клиент отправляет пакет:

```json
{
  "version": 1,
  "type": "register",
  "nickname": "alice",
  "public_key": "-----BEGIN PUBLIC KEY----- ... -----END PUBLIC KEY-----"
}
```

Сервер:

- проверяет уникальность никнейма;
- сохраняет `nickname -> {conn, public_key}`;
- отвечает `register_ok` или `error`.

Успешный ответ:

```json
{
  "version": 1,
  "type": "register_ok",
  "nickname": "alice"
}
```

## 5) Обмен публичными ключами

Когда `alice` хочет писать `bob`, ей нужен публичный ключ `bob`.

`alice` отправляет:

```json
{
  "version": 1,
  "type": "request_key",
  "nickname": "bob"
}
```

Сервер:

- если `bob` найден, возвращает `public_key`;
- если нет, отправляет `event` с `user_not_found`.

Ответ с ключом:

```json
{
  "version": 1,
  "type": "public_key",
  "nickname": "bob",
  "public_key": "-----BEGIN PUBLIC KEY----- ... -----END PUBLIC KEY-----"
}
```

Клиент сохраняет ключ локально в памяти (`_peer_keys`).

## 6) Отправка сообщения (подробный поток)

Пример: `alice` отправляет plaintext `"hello"` пользователю `bob`.

### Шаг 1. Подготовка у отправителя

Клиент `alice`:

1. берёт публичный ключ `bob`;
2. генерирует случайный AES-ключ (256 бит);
3. генерирует случайный `nonce` (12 байт);
4. шифрует plaintext через AES-GCM → `ciphertext`;
5. шифрует AES-ключ через RSA-OAEP (SHA-256) ключом `bob` → `encrypted_key`;
6. формирует `message_id` (UUID).

### Шаг 2. Передача на сервер

`alice` отправляет пакет:

```json
{
  "version": 1,
  "type": "chat_message",
  "message_id": "uuid",
  "from": "alice",
  "to": "bob",
  "encrypted_key": "base64...",
  "nonce": "base64...",
  "ciphertext": "base64..."
}
```

Поля бинарных данных (`encrypted_key`, `nonce`, `ciphertext`) передаются как Base64-строки.

### Шаг 3. Маршрутизация сервером

Сервер:

- проверяет, что `from` совпадает с зарегистрированным ником соединения;
- находит активное соединение `bob`;
- пересылает пакет `chat_message` получателю **без изменений**;
- отправителю возвращает `delivery_receipt`.

### Шаг 4. Расшифрование у получателя

Клиент `bob`:

1. Base64 → bytes;
2. своим RSA-приватным ключом расшифровывает `encrypted_key` и получает AES-ключ;
3. через AES-GCM расшифровывает `ciphertext` с `nonce`;
4. получает исходный plaintext.

## 7) Уведомления и служебные события

В проекте сервер отправляет события (`type: event`), например:

- `user_online`
- `user_offline`
- `user_not_found`

Также есть `delivery_receipt`:

```json
{
  "version": 1,
  "type": "delivery_receipt",
  "message_id": "uuid",
  "to": "bob"
}
```

И ошибки:

```json
{
  "version": 1,
  "type": "error",
  "code": "protocol_error",
  "message": "..."
}
```

## 8) Что сервер видит и чего не видит

Сервер видит:

- кто подключён;
- ник отправителя и получателя;
- факт и время отправки;
- зашифрованные поля (`encrypted_key`, `nonce`, `ciphertext`).

Сервер не видит:

- plaintext сообщения;
- приватные ключи клиентов.

## 9) Ограничения текущей реализации

1. Нет проверки подлинности публичного ключа (нет сертификатов/подписей).
2. Нет TLS между клиентом и сервером.
3. Ключи генерируются при запуске клиента и хранятся в памяти (без постоянного безопасного хранилища).
