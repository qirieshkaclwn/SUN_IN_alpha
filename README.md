# SUN_IN_alpha

TCP-чат с E2E-шифрованием и PKI-авторизацией клиентов (сертификаты, challenge-response).

## Что реализовано

- Приватные сообщения шифруются на клиенте и расшифровываются только на клиенте-получателе.
- Сервер не расшифровывает `text`, а только маршрутизирует пакет.
- Клиенты доверяют только сертификатам, подписанным `private_ca.crt`.
- Авторизация клиента: `auth_init -> auth_challenge -> auth_proof -> auth_success`.
- Автовыпуск сертификата: если у клиента нет `*.crt/*.key`, клиент сам генерирует ключ + CSR и запрашивает сертификат у сервера.

## Структура проекта

- `server/main.py` — сервер (роутинг, CA-подпись CSR, проверка сертификатов и подписи challenge)
- `client/main.py` — клиент (выпуск CSR, авторизация, E2E-шифрование/дешифрование)
- `server/private_ca.crt` — публичный сертификат CA
- `server/private_ca.key` — приватный ключ CA (секрет)
- `client/private_ca.crt` — pinned CA для проверки сертификатов

## Протокол передачи

Каждый пакет передается как:
1. 4 байта заголовка (длина JSON в big-endian)
2. JSON payload UTF-8

Базовые поля пакета:

```json
{
  "version": 1,
  "type": "packet_type",
  "timestamp": 1234567890,
  "from": "sender",
  "to": "recipient",
  "text": "...",
  "nickname": "q1",
  "client_cert": "-----BEGIN CERTIFICATE-----...",
  "csr": "-----BEGIN CERTIFICATE REQUEST-----...",
  "enc_key": "base64...",
  "nonce": "base64...",
  "signature": "base64...",
  "event": "auth_success",
  "error": "error text"
}
```

## Полный процесс работы

### 1) Подключение

Клиент открывает TCP-соединение с сервером `host:port`.

### 2) Проверка локальной identity клиента

Клиент ищет:
- `<nickname>.crt` + `<nickname>.key`, или
- `client.crt` + `client.key` (fallback).

Если найдено — загружает и проверяет:
- подпись сертификата от `private_ca.crt`,
- срок действия,
- `CN == nickname`,
- соответствие сертификата приватному ключу.

### 3) Автовыпуск сертификата (если файлов нет)

Клиент:
1. генерирует RSA-ключ,
2. создает CSR с `CN=nickname`,
3. отправляет `cert_enroll` (`nickname`, `csr`).

Сервер:
1. проверяет входные данные,
2. подписывает CSR своим `private_ca.key`,
3. отправляет `cert_enroll_response` с `client_cert`.

Клиент:
1. валидирует выданный сертификат,
2. сохраняет `<nickname>.crt` и `<nickname>.key`.

### 4) Авторизация клиента (challenge-response)

1. Клиент отправляет `auth_init` (`nickname`, `client_cert`).
2. Сервер проверяет сертификат (CA, срок, CN).
3. Сервер отправляет `auth_challenge` с `nonce`.
4. Клиент подписывает `nonce` своим приватным ключом и отправляет `auth_proof` (`signature`).
5. Сервер проверяет подпись публичным ключом из сертификата.
6. Сервер отправляет `event: auth_success`.

После этого клиент считается авторизованным.

### 5) Получение ключа собеседника

1. Если у отправителя нет ключа/сертификата получателя — отправляется `key_request`.
2. Сервер возвращает `key_response` с `client_cert` получателя.
3. Отправитель валидирует сертификат получателя (CA, срок, CN).
4. Извлекает публичный ключ получателя (см. подробно в разделе [Проверка сертификатов конечных пользователей](#проверка-сертификатов-конечных-пользователей)).


### 6) E2E-шифрование и отправка сообщения

На клиенте-отправителе:
1. генерируется случайный AES-ключ,
2. сообщение шифруется `AES-GCM` (`text` = base64 ciphertext),
3. AES-ключ шифруется RSA-публичным ключом получателя (`enc_key`),
4. формируется пакет `message` (`to`, `text`, `enc_key`, `nonce`).


На сервере:
- сервер проверяет формат и пересылает пакет получателю без расшифровки.

На клиенте-получателе:
1. RSA-приватным ключом расшифровывается `enc_key`,
2. `AES-GCM` расшифровывает `text`,
3. показывается исходный plaintext.

## Гибридное шифрование: подробное объяснение

### Почему гибридное (RSA + AES)?

**Проблема с чистым RSA:**
- RSA медленный (криптографические вычисления сложные)
- Можно зашифровать максимум ~200 байт (для RSA-2048)
- Для больших сообщений нужен другой подход

**Решение: гибридное шифрование**
- Большие данные шифруются **быстрым AES**
- AES-ключ защищается **медленным RSA**

---

### Схема шифрования: пошагово

#### **Шаг 1: Клиент q1 хочет отправить сообщение клиенту q2**

```
q1 (отправитель)
├─ Своё имя: q1
├─ Свой приватный ключ: q1_private.key (только у q1!)
└─ Публичный ключ q2: q2_public.key (получил ранее)

q2 (получатель)
└─ Публичный ключ: q2_public.key (видит q1)
   Приватный ключ: q2_private.key (только у q2!)
```

---

#### **Шаг 2: q1 генерирует случайный AES-ключ**

```python
import os
aes_key = os.urandom(32)  # 32 байта = 256 бит (AES-256)
# Результат: b'\x7f\x2a\xc3\x91...' (случайные байты)
```

**Зачем случайный?** Каждое сообщение используeт свой ключ. Безопаснее.

---

#### **Шаг 3: q1 шифрует текст сообщения AES-GCM**

```python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM

# Исходное сообщение
plaintext = "Привет q2, это конфиденциально!"

# Генерируем nonce (вектор инициализации) для AES-GCM
nonce = os.urandom(12)  # 12 байт для AES-GCM
# Результат: b'\x3c\x5b\x9e\x22...'

# Шифруем
aesgcm = AESGCM(aes_key)
ciphertext = aesgcm.encrypt(nonce, plaintext.encode('utf-8'), None)
# Результат: b'\x8c\x5f\x11\x2b\x3a\x4d...' (зашифрованные байты)
```

**Что произошло:**
- ✓ `plaintext` (`"Привет q2..."`) → `ciphertext` (невразумительные байты)
- ✓ Без `aes_key` и `nonce` никто не может расшифровать
- ✓ Даже если перехватить, это бесполезно

**Отправляем на Шаге 5:**
```json
{
  "text": "jMXxEiwbOqNdJk2x...",  // base64(зашифрованное сообщение)
  "nonce": "PFuePi4cKj9xM2..."   // base64(nonce) — нужен для расшифровки текста
}
```

---

#### **Шаг 4: q1 шифрует AES-ключ публичным ключом q2 (RSA)**

```python
from cryptography.hazmat.primitives.asymmetric import padding
from cryptography.hazmat.primitives import hashes

# AES-ключ, который нужно защитить
aes_key = b'\x7f\x2a\xc3\x91...'  # 32 байта

# Публичный ключ q2 (получен ранее из сертификата)
q2_public_key = cert.public_key()  # RSA-2048

# Шифруем AES-ключ RSA-методом OAEP
encrypted_aes_key = q2_public_key.encrypt(
    aes_key,
    padding.OAEP(
        mgf=padding.MGF1(algorithm=hashes.SHA256()),
        algorithm=hashes.SHA256(),
        label=None
    )
)
# Результат: b'\x9c\x3f\x5d\x8a\x2b...' (256 байт для RSA-2048)
```

**Что произошло:**
- ✓ AES-ключ (32 байта) → зашифрованный AES-ключ (256 байт для RSA-2048)
- ✓ Только q2 может расшифровать (у него есть q2_private.key)
- ✓ q1 не может расшифровать свой же зашифрованный ключ

**⚠️ ВАЖНО — что отправляем на сервер:**
```json
{
  "enc_key": "nDPdijK2wx...",        // base64(зашифрованный AES-ключ RSA!)
  "text": "jMXxEiwbOqNdJk2x...",      // base64(зашифрованное сообщение AES!)
  "nonce": "PFuePi4cKj9xM2..."       // base64(nonce для AES)
}
```

**Расшифровка возможна ТОЛЬКО у q2:**
- `text` может расшифровать только кто знает `aes_key`
- `aes_key` может расшифровать только кто имеет `q2_private_key`
- `q2_private_key` есть только у q2!
- **Сервер видит ВСЕ три поля, но не может расшифровать ни один!**

---

#### **Шаг 5: Сервер пересылает пакет q2**

Сервер **не трогает** шифрованные данные:
```json
{
  "type": "message",
  "from": "q1",
  "to": "q2",
  "text": "jMXxEiwbOqNdJk2x...",      // ← зашифрованное сообщение (AES)
  "enc_key": "nDPdijK2wx...",   // ← зашифрованный ключ (RSA)
  "nonce": "PFuePi4cKj9xM2..."  // ← nonce для расшифровки (открытый)
}
```

**Кто может расшифровать?**
```
┌─────────────────────────────────────────────────────┐
│ Видит пакет на сервере                              │
│                                                     │
│ text (зашифрованное сообщение)   ← Только q2      │
│ enc_key (зашифрованный ключ)     ← Только q2      │
│ nonce (открытый)                 ← Все видят      │
│                                                     │
│ Сервер видит ВСЕ три поля,                         │
│ но расшифровать не может!                          │
└─────────────────────────────────────────────────────┘
```

---

#### **Шаг 6: q2 получает пакет и расшифровывает**

**Этап 1: Расшифровать AES-ключ (RSA)**

```python
# Зашифрованный AES-ключ из пакета
encrypted_aes_key = base64.b64decode("nDPdijK2wx...")

# Приватный ключ q2 (только у q2!)
q2_private_key = load_private_key("q2.key")

# Расшифровываем
aes_key = q2_private_key.decrypt(
    encrypted_aes_key,
    padding.OAEP(
        mgf=padding.MGF1(algorithm=hashes.SHA256()),
        algorithm=hashes.SHA256(),
        label=None
    )
)
# Результат: b'\x7f\x2a\xc3\x91...' (тот же ключ, что генерировал q1!)
```

**Что произошло:**
- ✓ Только q2 может расшифровать (у неё есть приватный ключ)
- ✓ q1 не может расшифровать (у неё нет q2_private.key)

---

**Этап 2: Расшифровать текст (AES-GCM)**

```python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM

# Восстановленный AES-ключ
aes_key = b'\x7f\x2a\xc3\x91...'

# Зашифрованные данные и nonce из пакета
ciphertext = base64.b64decode("jMXxEiwbOqNdJk2x...")
nonce = base64.b64decode("PFuePi4cKj9xM2...")

# Расшифровываем
aesgcm = AESGCM(aes_key)
plaintext = aesgcm.decrypt(nonce, ciphertext, None)
# Результат: b'Привет q2, это конфиденциально!'

# Выводим
print(plaintext.decode('utf-8'))
# ✓ Привет q2, это конфиденциально!
```

**Что произошло:**
- ✓ `ciphertext` → `plaintext` (восстановлено исходное сообщение)
- ✓ Без правильного `aes_key` и `nonce` расшифровать невозможно

```
┌─────────────────────────────────────────────────────────────────┐
│ q1 (ОТПРАВИТЕЛЬ)                                                │
├─────────────────────────────────────────────────────────────────┤
│ 1. plaintext = "Привет q2..."                                   │
│ 2. aes_key = random(32 байта)                                   │

│ 3. plaintext = AES_GCM_decrypt(text, aes_key, nonce)            │

└─────────────────────────────────────────────────────────────────┘
```

## Типы пакетов

- `cert_enroll` — запрос на выпуск сертификата по CSR
- `cert_enroll_response` — выданный сертификат клиента
- `auth_init` — старт авторизации
- `auth_challenge` — nonce от сервера
- `auth_proof` — подпись nonce
- `key_request` — запрос сертификата пользователя
- `key_response` — сертификат пользователя
- `message` — E2E-сообщение
- `event` — системные события (`auth_success`, `user_joined`, `user_left`, `users_list`)
- `error` — ошибка протокола/валидации

## Работа с сертификатами и PKI

### Инфраструктура ключей (PKI)

Система использует **самоподписанный приватный CA (Certificate Authority)** для выпуска клиентских сертификатов:

- `server/private_ca.key` — приватный ключ CA (хранится только на сервере, **никогда не передается**)
- `server/private_ca.crt` — публичный сертификат CA (копируется на каждого клиента)
- `client/private_ca.crt` — pinned копия CA на клиенте (для проверки подписей)

**Почему это безопасно:**
- Клиент не может подделать сертификат, т.к. у него нет приватного ключа CA
- Сервер может подписывать только корректные CSR (Certificate Signing Requests) с верным CN (Common Name = nickname)
- Даже если перехватить сертификат клиента — его невозможно подделать или модифицировать

### Жизненный цикл сертификата клиента

#### Выпуск сертификата (если клиент новый)

1. **Генерация ключевой пары (клиент):**
   - Клиент генерирует RSA-ключ (2048 бит)
   - Сохраняет в `<nickname>.key`

2. **Создание CSR (Certificate Signing Request):**
   - CSR содержит публичный ключ и CN=nickname
   - Подписан приватным ключом клиента
   - Гарантирует, что клиент владеет приватным ключом

3. **Отправка на сервер (`cert_enroll`):**
   ```json
   {
     "type": "cert_enroll",
     "nickname": "q1",
     "csr": "-----BEGIN CERTIFICATE REQUEST-----\n..."
   }
   ```

4. **Подпись сервером:**
   - Сервер проверяет CN в CSR совпадает с nickname
   - Подписывает CSR своим приватным ключом CA
   - Выпускает сертификат на 365 дней
   - Отправляет обратно клиенту

5. **Сохранение (клиент):**
   - Клиент валидирует выданный сертификат:
     - Проверяет подпись от pinned CA
     - Проверяет срок действия
     - Проверяет CN == nickname
     - Проверяет соответствие публичного ключа приватному
   - Сохраняет в `<nickname>.crt`

#### Загрузка локального сертификата (последующие запуски)

При каждом подключении клиент сначала проверяет наличие локального сертификата:

```
Поиск: <nickname>.crt + <nickname>.key
Fallback: client.crt + client.key
```

Валидация перед использованием:
- ✓ Подпись сертификата от CA
- ✓ Срок действия (не истек, еще не начал действовать)
- ✓ CN в сертификате == nickname
- ✓ Соответствие публичного ключа в сертификате приватному ключу

### Авторизация клиента (Challenge-Response)

После загрузки сертификата клиент проходит **challenge-response авторизацию**:

1. **Инициация (`auth_init`):**
   ```
   Клиент → Сервер: nickname, client_cert
   ```

2. **Вызов (`auth_challenge`):**
   - Сервер валидирует сертификат (CA, срок, CN)
   - Генерирует случайный nonce (32 байта)
   - Отправляет клиенту: `nonce`

3. **Доказательство (`auth_proof`):**
   - Клиент подписывает `nonce` своим приватным ключом (PKCS#1 v1.5 + SHA-256)
   - Отправляет сервису: `signature`
   ```
   Клиент → Сервер: signature = RSA_sign(nonce, private_key)
   ```

4. **Верификация (сервер):**
   - Извлекает публичный ключ из сертификата клиента
   - Проверяет подпись: `RSA_verify(signature, nonce, public_key) == OK`
   - Если верно → отправляет `event: auth_success`

**Почему это защищено:**
- Nonce одноразовый → защита от replay-атак
- Только владелец приватного ключа может подписать nonce
- Сервер проверяет подпись публичным ключом из сертификата
- Если сертификат поддельный → подпись не пройдет проверку

### Проверка сертификатов конечных пользователей

При обмене сообщениями клиент должен получить **публичный ключ получателя**:

1. **Запрос (`key_request`):**
   ```
   Клиент A → Сервер: to=q2
   ```

2. **Ответ (`key_response`):**
   ```
   Сервер → Клиент A: nickname=q2, client_cert=<q2.crt>
   ```

3. **Валидация (клиент A):**
   - Проверяет подпись сертификата от pinned CA
   - Проверяет срок действия
   - Проверяет CN == nickname
   - **Извлекает публичный ключ из сертификата:**
     ```python
     # Загрузить сертификат из PEM
     cert = x509.load_pem_x509_certificate(cert_pem.encode('utf-8'))
     
     # Извлечь публичный ключ
     public_key = cert.public_key()
     # public_key теперь содержит RSA-ключ из сертификата q2
     ```
   - Сохраняет публичный ключ в памяти (dict):
     ```python
     self.peer_public_keys[nickname] = public_key
     # Теперь можем шифровать для q2 без повторных запросов
     ```

**Как работает извлечение публичного ключа:**

1. Сертификат **содержит встроенный публичный ключ** в формате X.509
2. Метод `.public_key()` распаковывает публичный ключ из структуры сертификата
3. Ключ может быть разного типа (RSA, EC, Ed25519, Ed448)
4. Для шифрования нам нужен именно **публичный ключ** (приватный есть только у q2)
5. Ключ кэшируется в памяти, чтобы не запрашивать его снова

**Структура сертификата X.509:**
```
┌─ Subject: CN=q2, O=SUN_IN, C=RU
├─ Issuer: CN=SUN_IN_CA, O=SUN_IN, C=RU
├─ Public Key (RSA 2048)  ← Вот это извлекается!
├─ Validity: 2026-04-16 ... 2027-04-16
├─ Serial Number: 12345...
├─ Signature (CA's signature)
└─ ...другие поля...
```


### Параметры сертификатов
